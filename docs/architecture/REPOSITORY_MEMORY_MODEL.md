# Repository Memory Model

## Purpose

This document defines the first-class Repository Memory System architecture for Playbook.

Rule: Keep CLI-first, local-first, private-first operation intact.
Rule: Preserve the existing structural graph contract as-is.

## Layered memory architecture

Playbook memory is modeled as five deterministic layers:

1. **Structural intelligence (current deterministic substrate)**
   - `.playbook/repo-index.json`
   - `.playbook/repo-graph.json`
   - Produced by existing repository intelligence commands (`index`, `graph`, `query`, `explain`, `ask --repo-context`).
   - This layer captures repository structure and governed relationships, not temporal workflow history.

2. **Working context (short-horizon execution context)**
   - `.playbook/context/*`
   - Includes module digests, scoped context bundles, and task-oriented snapshots generated by current command flows.
   - Intended for fast local execution continuity and token-efficient read runtime behavior.

3. **Episodic memory (time-ordered repository events)**
   - `.playbook/memory/events/*.json`
   - `.playbook/memory/index.json`
   - Captures deterministic event records (verify outcomes, remediation attempts, approvals, review outcomes, recurring findings) with provenance.

4. **Replay/consolidation layer (human-reviewed synthesis queue)**
   - `.playbook/memory/candidates.json`
   - Stores replay-ready candidate consolidations derived from episodic memory and structural context.
   - This layer is review-oriented and must not autonomously mutate repository governance or source files.

5. **Doctrine/policy memory (durable governed knowledge)**
   - Rules, contracts, docs, remediation templates, and promoted governance artifacts.
   - Durable memory requires explicit promotion, reviewability, and contract alignment before becoming doctrine.

## Graph boundary clarification

`.playbook/repo-graph.json` remains the **structural repository graph contract**.

It must not be repurposed as a full temporal memory database or event log.

Pattern: Structural Graph + Memory Graph/Index.

The structural graph continues to model deterministic code/repository relationships, while temporal memory uses separate event/index artifacts under `.playbook/memory/*`.

## Promotion and replay rules

Rule: Promotion Required for Durable Doctrine.
Rule: Replay Is Human-Review-Oriented, Not Autonomous Mutation.

Pattern: Fast Episodic Store, Slow Doctrine Store.
Rule: Canonical governance artifacts are preserved; everything else earns retention under pressure.
Pattern: Admit -> dedupe -> compact -> summarize -> evict.
Failure Mode: Hard-cap or random deletion policies destabilize memory and destroy useful context.

- Episodic artifacts may update quickly as commands execute.
- Doctrine artifacts update slowly and only through explicit review/promotion workflows.

## Failure modes to avoid

Failure Mode: Memory Hoarding.

- Capturing all events without compaction/pruning reduces retrieval quality and erodes trust.

Failure Mode: Rebuilding Durable Memory From Current Repo State Only.

- Durable doctrine cannot be reconstructed solely from current files without event lineage, review history, and promotion evidence.
